# 📝 Gartner AI Translator 개발 및 고도화 기록 (Decision Record)

본 문서는 서비스의 초기 단계부터 현재의 현대적인 아키텍처로 진화하기까지의 주요 의사결정과 기술적 변화를 기록합니다.

---

## 📅 프로젝트 여정

### 1단계: Streamlit 기반 초기 프로토타입 (Monolithic)

- **목표:** 빠르게 동작하는 PPT 번역 기능 구현.
- **특징:** Python Streamlit을 사용하여 UI와 로직을 하나로 관리.
- **한계:** 세션 관리의 복잡성, UI 커스터마이징의 제약, 대규모 사용자 대응을 위한 확장성 부족.

### 2단계: 아키텍처 전환 (FastAPI + React Decoupling)

- **의사결정:** 서비스의 확장성과 전문적인 UI를 위해 **FastAPI(Back)**와 **React(Front)**로 구조를 완전히 분리.
- **기술적 이점:**
  - 비동기 처리(asyncio)를 통한 파일 처리 성능 향상.
  - WebSocket을 사용한 실시간 번역 진행률 및 로그 스트리밍.
  - Tailwind CSS를 활용한 현대적이고 미려한 디자인 구축.

### 3단계: 주요 기능 고도화

- **다국어 지원(i18n):** 글로벌 오피스 대응을 위해 한국어, 영어, 일본어 UI 전환 시스템 구축.
- **영구 저장소(History):** SQLite를 도입하여 사용자가 과거에 번역한 파일을 언제든 다시 다운로드할 수 있도록 이력 관리 시스템 구현.
- **테스트 모드:** 전체 번역 전 결과물을 미리 확인할 수 있도록 10페이지 제한 번역 로직 추가.
- **가이드 챗봇:** 사용자 경험 향상을 위해 규칙 기반의 도움말 챗봇 위젯 탑재.

### 4단계: 컨테이너화 및 구조 최적화

- **Docker Compose:** 한 줄의 명령어로 백엔드와 프론트엔드를 동시에 배포할 수 있는 환경 구축.
- **폴더 구조 재편 (현재):** `backend`, `frontend`, `output`으로 역할을 명확히 분리하고 사용하지 않는 레거시 코드를 정리하여 유지보수성 극대화.

---

## 🧠 핵심 기술 결정 (Key Decisions)

1. **서버 측 문서 처리 (Server-Side Processing)**
   - 브라우저 직접 호출 방식은 API Key 노출 위험이 있고 복잡한 PPT/PDF 레이아웃을 유지하기 위한 라이브러리(python-pptx, PyMuPDF) 활용이 제한적이므로, 모든 파일 처리는 보안이 확보된 서버에서 수행하기로 결정.
2. **용어집(Glossary) 강제 적용**
   - 사내 전문 용어의 일관성을 위해 AI 번역 전후에 정규표현식을 사용한 용어 치환 로직을 설계하여 번역 품질의 일관성을 확보.
3. **UV 패키지 매니저 사용**
   - 기존 `pip`보다 수십 배 빠른 설치 속도와 엄격한 종속성 관리를 위해 `uv`를 백엔드 패키지 매니저로 채택.
